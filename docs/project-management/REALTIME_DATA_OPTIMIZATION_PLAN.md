# 실시간 데이터 업데이트 최적화 계획

## 목표
30초마다 실시간 데이터를 업데이트하면서도 시스템 부하를 최소화

## 현재 문제점

1. **비효율적인 데이터 플로우**: 백엔드는 6분마다 수집하지만 프론트엔드는 30초마다 API 호출 (불필요한 호출 91.7%)
2. **캐싱 미구현**: 매 요청마다 DB 쿼리 실행
3. **React Query 설정 불일치**: staleTime 5분이지만 자동 갱신은 30초

## 개선 방안

### 계층적 캐싱 + 최적화된 수집 주기

```
외부 소스 (네이버, 한국거래소)
    ↓ 1분마다 수집
데이터베이스
    ↓ 캐시 미스 시에만 쿼리
메모리 캐시 (30초 TTL)
    ↓ 30초마다 최신 데이터 제공
프론트엔드 (React Query)
```

## 구현 계획

### Phase 1: 백엔드 캐싱 (우선순위: 높음)

1. **메모리 캐시 구현** (`backend/app/utils/cache.py`)
   - TTL 기반, 스레드 안전성, 통계 제공

2. **API 라우터에 캐싱 적용** (`etfs.py`, `data.py`, `news.py`)
   - 캐시 키: `{endpoint}:{ticker}:{params_hash}`
   - TTL: 30초

### Phase 2: 스케줄러 최적화 (우선순위: 높음)

1. **수집 주기 조정**
   - `SCRAPING_INTERVAL_MINUTES`: 6분 → 1분
   - `CACHE_TTL_MINUTES`: 3분 → 0.5분 (30초)

2. **로깅 개선**: 수집 소요 시간, 캐시 히트율, 재시도 로직

### Phase 3: 프론트엔드 최적화 (우선순위: 중간)

1. **React Query 설정**: `staleTime: 30000` (30초로 통일)
2. **자동 갱신**: 기본 30초 유지

### Phase 4: 모니터링 (우선순위: 중간)

1. **캐시 통계 API**: `GET /api/data/cache/stats`
2. **수집 상태 표시**: 마지막 수집 시간, 다음 예정 시간

## 성능 개선 효과

| 항목 | 현재 | 개선 후 | 효과 |
|------|------|---------|------|
| 시간당 DB 쿼리 | 120회 | 60회 | 50% 감소 |
| 응답 속도 | ~20ms | ~1ms | 80% 향상 |
| 데이터 신선도 | 최대 6분 지연 | 최대 1분 지연 | 600% 개선 |
| 외부 API 호출 | 10회 | 60회 | 증가 (안전 수준) |

## 구현 순서

### Sprint 1: 핵심 기능 (2-3시간)
✅ 메모리 캐시 구현
✅ ETF API 캐싱 적용
✅ 환경 변수 업데이트
✅ 기본 테스트

### Sprint 2: 전체 적용 (1-2시간)
✅ 모든 API 캐싱 적용
✅ React Query 설정 조정
✅ 캐시 통계 API
✅ 통합 테스트

### Sprint 3: 모니터링 (1시간)
✅ 수집 상태 표시 개선
✅ 로깅 개선
✅ 문서 업데이트

## 테스트

- **단위 테스트**: 캐시 동작, TTL, 스레드 안전성
- **통합 테스트**: 30초 자동 갱신, 캐시 히트/미스, 동시 요청
- **성능 테스트**: 응답 시간, 캐시 히트율, 메모리 사용량

## 위험 및 대응

1. **외부 API Rate Limit**: 필요시 수집 주기를 2-3분으로 조정
2. **메모리 사용량 증가**: `CACHE_MAX_SIZE` 제한, LRU eviction
3. **캐시 불일치**: 수동 수집 시 캐시 무효화 API 제공

## 성공 기준

### 필수
✅ 30초마다 자동 갱신
✅ 1분 이내 최신 데이터
✅ 응답 시간 50ms 이하
✅ DB 쿼리 50% 감소

### 선택
✅ 캐시 히트율 50% 이상
✅ 통계 대시보드
✅ 수집 상태 표시
