# Definition of Done (완료 기준)

## 개요

이 문서는 프로젝트의 모든 작업이 "완료"로 간주되기 위한 기준을 정의합니다.
**모든 기준을 충족하지 않으면 다음 단계로 진행할 수 없습니다.**

---

## 🔒 핵심 원칙

### 원칙 1: 테스트 우선 (Test-First)
```
✅ 모든 기능은 100% 테스트 완료 후 다음 단계로 진행
❌ 테스트 없이 다음 기능 개발 금지
```

### 원칙 2: 단계별 검증 (Phase Gate)
```
각 Phase는 반드시 Acceptance Criteria를 모두 충족해야 다음 Phase 시작 가능
```

### 원칙 3: 문서화 필수 (Documentation Required)
```
코드 변경 시 관련 문서 동시 업데이트
```

---

## ✅ 전체 작업의 Definition of Done

모든 작업(Task, Feature, Phase)은 다음을 만족해야 "완료"로 간주됩니다:

### 1. 코드 품질
- [ ] 코드가 작성되고 동작함
- [ ] 린트 에러 없음 (ESLint, Pylint)
- [ ] 코드 리뷰 완료 (팀 프로젝트의 경우)
- [ ] 코딩 컨벤션 준수 (DEVELOPMENT_GUIDE.md 참조)

### 2. 테스트 (필수)
- [ ] **유닛 테스트 작성 및 100% 통과**
- [ ] **통합 테스트 작성 및 통과** (API 엔드포인트의 경우)
- [ ] 엣지 케이스 테스트 포함
- [ ] 에러 핸들링 테스트 포함

### 3. 문서화
- [ ] 코드 주석 작성 (복잡한 로직)
- [ ] API 변경 시 API_SPECIFICATION.md 업데이트
- [ ] DB 변경 시 DATABASE_SCHEMA.md 업데이트
- [ ] README 또는 관련 문서 업데이트

### 4. 검증
- [ ] 로컬 환경에서 정상 동작 확인
- [ ] 브라우저 테스트 (프론트엔드의 경우)
- [ ] 성능 저하 없음 확인

### 5. 정리
- [ ] 디버깅용 코드 제거 (console.log, print 등)
- [ ] 불필요한 주석 제거
- [ ] TODO 주석 처리 또는 이슈 생성
- [ ] Git 커밋 메시지 규칙 준수

## 🧪 테스트 정책

### 테스트 커버리지 목표
- **백엔드**: 최소 80% 코드 커버리지
- **프론트엔드**: 최소 70% 코드 커버리지
- **Critical Path**: 100% 커버리지 (데이터 수집, API 엔드포인트)

### 테스트 실행 시점
1. **개발 중**: 기능 구현 후 즉시 테스트 작성 및 실행
2. **커밋 전**: 모든 테스트 통과 확인
3. **PR/MR 전**: 전체 테스트 스위트 실행
4. **배포 전**: 통합 테스트 + E2E 테스트

## 📊 완료 체크리스트

각 Phase 완료 시 다음을 확인:

```bash
# 백엔드 테스트 실행
cd backend
pytest tests/ -v --cov=app --cov-report=html

# 프론트엔드 테스트 실행
cd frontend
npm test -- --coverage

# 린트 체크
# 백엔드
pylint app/

# 프론트엔드
npm run lint
```

**모든 명령이 에러 없이 통과해야 다음 Phase 진행 가능**

## 📝 체크리스트 사용 방법

### 작업 시작 시
1. 해당 Phase의 DoD를 읽음
2. 기능 구현 + 테스트 작성 계획 수립

### 작업 중
- [ ] 기능 구현
- [ ] 테스트 작성
- [ ] 테스트 100% 통과
- [ ] 린트 체크
- [ ] 문서 업데이트

### 작업 완료 시
1. 이 문서의 체크리스트 확인
2. TODO.md 체크박스 체크
3. PROGRESS.md에 기록
4. 다음 Phase 시작
