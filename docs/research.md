# ETFWeeklyReport 프로젝트 심층 연구 보고서

이 보고서는 ETFWeeklyReport 프로젝트의 코드베이스를 심층 분석한 결과를 바탕으로 작성되었습니다. 본 프로젝트는 한국 ETF 시장의 데이터를 수집, 분석, 시각화하고 AI 에이전트와 연동할 수 있는 종합 금융 플랫폼입니다.

---

## 1. 시스템 아키텍처 및 데이터 흐름

### 1.1 전체 구조
프로젝트는 현대적인 4계층 아키텍처로 구성되어 있습니다.
- **Frontend**: React (Vite) 기반의 SPA (Single Page Application)
- **Backend**: FastAPI 기반의 비동기 REST API 서버
- **Database**: SQLite (개발/기본) 및 PostgreSQL (운영) 지원
- **Desktop & MCP**: Electron 패키징 및 Model Context Protocol 서버 제공

### 1.2 데이터 흐름 (Data Flow)
1. **수집**: `ETFDataCollector`가 Naver Finance 및 Naver Search API로부터 시세, 매매동향, 뉴스를 스크래핑합니다.
2. **저장**: 수집된 데이터는 검증(Validation) 및 정제(Cleaning) 과정을 거쳐 RDBMS에 저장됩니다.
3. **가공**: 백엔드 서비스 계층에서 수익률, 변동성, 샤프 지수 등 주요 메트릭을 계산합니다.
4. **소비**: 프런트엔드(React)는 차트와 테이블로 시각화하고, MCP 서버는 AI 에이전트(Claude 등)가 활용할 수 있는 도구(Tools)로 제공합니다.

---

## 2. 백엔드 (Backend) 기술 분석

### 2.1 데이터 수집 전략 (Smart Scraper)
`backend/app/services/data_collector.py`는 본 프로젝트의 핵심 엔진입니다.
- **중복 방지 (Smart Collection)**: 데이터베이스의 마지막 기록 날짜를 확인하여 누락된 기간만 선택적으로 수집합니다.
- **병렬 처리**: `ThreadPoolExecutor`를 사용하여 여러 종목의 데이터를 동시에 수집하여 성능을 극대화합니다.
- **안정성**: `RateLimiter`로 요청 속도를 제어하고, `retry_with_backoff` 데코레이터를 통해 네트워크 오류에 대응합니다.

### 2.2 API 설계 및 모델링
- **Pydantic V2**: 강력한 데이터 검증 및 스키마 정의를 위해 Pydantic을 전방위적으로 사용합니다.
- **비동기 처리**: FastAPI의 비동기 기능을 활용하여 효율적인 IO 처리를 수행합니다.
- **통합 로깅**: `structlog`를 사용하여 구조화된 로그를 남겨 운영 편의성을 높였습니다.

## 3. 프런트엔드 (Frontend) 및 시각화

### 3.1 기술 스택
- **React 18**: 컴포넌트 기반 UI 개발.
- **Tailwind CSS**: 유틸리티 우선 방식의 빠른 스타일링.
- **Recharts**: 금융 차트(캔들스틱, 선 그래프, 히트맵 등) 시각화의 핵심 라이브러리.
- **TanStack Query (v5)**: 서버 상태 관리 및 캐싱을 효율적으로 처리.

### 3.2 주요 UI 구성 요소
- **Dashboard**: 관심 종목의 주간 수익률 및 최신 뉴스를 한눈에 파악.
- **ETF Detail**: 상세 시세 차트, 매매동향(외인/기관), AI 인사이트 요약 제공.
- **Scanner**: 필터 조건을 조합하여 투자 대상 ETF를 발굴하는 강력한 도구.

## 4. MCP (Model Context Protocol) 서버

`mcp-server/`는 이 프로젝트를 단순한 웹 앱을 넘어 AI 생태계로 확장시키는 핵심 요소입니다.
- **도구 노출 (Tool Exposure)**: `list_stocks`, `get_etf_prices`, `simulate_portfolio` 등 15개 이상의 백엔드 기능을 MCP 도구로 변환합니다.
- **AI 연동**: Claude와 같은 LLM이 직접 실시간 시장 데이터를 조회하고 투자 시뮬레이션을 실행하여 리포트를 작성할 수 있게 합니다.

## 7. 상세 구현 기능 및 지표 계산식

본 섹션에서는 프로그램 내에서 사용되는 주요 금융 지표의 계산 로직을 설명합니다. (최근 분석을 통해 표준 거래일 기준으로 수정되었습니다.)

### 7.1 수익률 계산식 (Returns)
모든 수익률은 퍼센트(%) 단위로 계산되며, `(현재가 - 기준가) / 기준가 * 100` 공식을 따릅니다.

1.  **일별 수익률 (`daily_change_pct`)**:
    -   공식: `(당일 종가 - 전일 종가) / 전일 종가 * 100`
    -   데이터 소스: 네이버 금융의 전일비를 파싱하여 활용.
2.  **주간 수익률 (`1w`)**:
    -   공식: `(현재가 - 5거래일 전 종가) / 5거래일 전 종가 * 100`
    -   수정 사항: 기존 7데이터 포인트에서 표준 영업일인 **5거래일**로 수정됨.
3.  **월간 수익률 (`1m`)**:
    -   공식: `(현재가 - 20거래일 전 종가) / 20거래일 전 종가 * 100`
    -   수정 사항: 기존 30데이터 포인트에서 표준 영업일인 **20거래일**로 수정됨.
4.  **연간 수익률 (`ytd`)**:
    -   공식: `(현재가 - 올해 첫 거래일 종가) / 올해 첫 거래일 종가 * 100`
    -   기준일: 해당 연도 1월 1일 이후의 가장 빠른 거래일.

### 7.2 스캐너 (`/scanner`) 주요 지표
스캐너 화면에서 제공되는 실시간 및 집계 데이터의 정의는 다음과 같습니다.

-   **현재가 (`close_price`)**: 수집 시점의 최신 종가.
-   **등락률 (`daily_change_pct`)**: 전일 대비 당일 가격 변동폭(%).
-   **거래량 (`volume`)**: 당일 누적 거래량 (단위: 주).
-   **주간 수익률 (`weekly_return`)**:
    -   공식: `(현재가 - 5거래일 전 종가) / 5거래일 전 종가 * 100`
    -   상세 페이지 및 스캐너 페이지 모두 동일하게 **최근 5번째 거래일(1주일 전)** 데이터를 기준으로 하는 롤링 윈도우(Rolling Window) 방식을 사용합니다.
-   **월간 수익률 (`monthly_return`)**:
    -   공식: `(현재가 - 20거래일 전 종가) / 20거래일 전 종가 * 100`
    -   스캐너에서 최근 **20거래일(약 1개월)** 성과를 파악하기 위해 사용됩니다.
-   **연간 수익률 (`ytd_return`)**:
    -   공식: `(현재가 - 올해 첫 거래일 종가) / 올해 첫 거래일 종가 * 100`
    -   올해 초 대비 현재까지의 성과를 나타냅니다.
-   **외국인/기관 순매수 (`foreign_net`, `institutional_net`)**:
    -   최근 1거래일 동안의 투자자별 순매수 수량입니다.
    -   단위: **주(Quantity)**. (네이버 금융의 투자자별 매매동향 데이터 기반)

### 7.3 거래일(Trading Day)의 정의
본 프로그램에서 사용하는 '5거래일' 또는 '20거래일'은 달력상의 일수가 아닌 **주식 시장이 실제로 개장된 날**만을 카운트합니다.
-   **주말 및 공휴일 제외**: 토/일요일 및 시장 휴장일은 데이터 포인트에서 제외되므로, 자동으로 다음(혹은 이전) 영업일 데이터를 참조합니다.
-   **데이터 무결성**: 수집 시점에 5개 미만의 데이터만 존재할 경우(예: 신규 상장주), 가용한 가장 오래된 데이터를 기준가로 사용하여 오차를 최소화합니다.

## 8. 작업 스케줄링 및 취소 흐름 (Bug Report)

시스템의 백그라운드 데이터 수집(Catalog Data Collection) 흐름을 심층 분석한 결과, **사용자의 취소 요청(Cancel Request)이 무시되고 백그라운드 작업이 계속 실행되는 심각한 리소스 누수(좀비 태스크) 버그**가 발견되었습니다.

### 8.1 버그 원인 분석
데이터 수집 작업(`CatalogDataCollector.collect_all`)은 4개의 페이즈(Phase)로 나뉘어 실행됩니다.
1. ETF 가격 수집
2. ETF 수급 수집 (병렬)
3. DB 저장
4. **KOSPI/KOSDAQ 주식 가격+수급 업데이트** (문제 발생 구간)

Phase 1~3에서는 작업 단위 사이에 `is_cancelled(TASK_ID)`를 확인하여 정상적으로 프로세스를 중단하지만, 가장 부하가 큰 **Phase 4(`_update_stock_prices`) 내부에는 취소 확인 로직이 전무**합니다.

### 8.2 상세 문제점
1. **페이지네이션 루프 내 취소 부재**: Phase 4 초기에는 코스피/코스닥 전체 목록을 가져오기 위해 네이버 금융을 최대 190페이지(80+110) 순회합니다. 이 `while` 루프 중간에 취소 요청이 들어와도 루프는 멈추지 않습니다.
2. **병렬 워커(ThreadPool) 통제 불능**: 상위 500개 주식의 수급 데이터를 모으기 위해 `ThreadPoolExecutor`를 사용하지만, 각 워커 스레드나 `as_completed` 처리 루프 안에서 `is_cancelled`를 검사하지 않습니다.
3. **Future 취소 로직 누락**: 취소 이벤트 발생 시 대기열에 있는 작업을 폐기하는 `future.cancel()` 처리가 없어, 한 번 스케줄링된 500개의 네트워크 요청이 무조건 끝까지 실행됩니다.

### 8.3 시스템에 미치는 영향
- **네트워크 차단 위험**: 스크래퍼가 멈추지 않고 불필요한 트래픽을 계속 유발하여 네이버 금융 서버로부터 IP가 차단(Rate Limit)될 수 있습니다.
- **서버 리소스 고갈**: 취소 후 새로운 수집 작업을 다시 시작할 경우, 기존의 좀비 스레드와 새로운 스레드가 동시에 실행되어 서버의 메모리와 CPU를 급격히 소모합니다.

이 문제는 시스템의 안정성을 크게 저해하므로 `CatalogDataCollector` 및 `TickerCatalogCollector`의 긴 루프 및 스레드 풀 내부에 명시적인 취소 확인 및 방어 로직을 즉각 주입해야 합니다.

## 9. 애널리스트 관점의 개선 및 추가 기능 제안 (초보자 중심)

본 섹션은 **주식을 처음 시작하는 초보 투자자**의 관점에서, 현재 구현된 기능의 한계와 추가되었으면 하는 기능을 주식 애널리스트 시각으로 분석한 결과입니다.

---

### 9.1 용어/개념 교육 기능 (가장 시급한 개선)

**현재 문제점**: RSI, MACD, MDD, 샤프지수, 순매수 등 전문 금융 용어가 초보자에게 아무런 설명 없이 노출됩니다. 차트와 숫자는 있지만 "그래서 내가 어떻게 해야 하는가?"에 대한 답이 없습니다.

**제안하는 개선:**
1. **용어 툴팁(Tooltip)**: 각 지표명 옆에 `?` 아이콘을 두고, 마우스 오버 시 "RSI가 30 이하이면 과매도 구간으로 반등 가능성이 높습니다" 같은 쉬운 설명을 제공합니다.
2. **지표 해석 배지(Badge)**: 숫자 옆에 "과매수", "과매도", "적정", "위험" 등 색상 배지를 함께 표시하여 수치를 직접 해석하지 않아도 상태를 파악할 수 있게 합니다.
   - 예: RSI 78 → 🔴 과매수 / MDD -25% → 🟡 주의
3. **온보딩 튜토리얼**: 첫 방문 시 각 화면의 주요 기능을 단계별로 안내하는 인터랙티브 투어(Guided Tour) 제공.
4. **용어 사전 페이지**: 앱 내 `/glossary` 페이지를 만들어 모든 금융 용어를 사전 형식으로 설명.

---

### 9.2 시장 맥락(Context) 정보 부재

**현재 문제점**: 현재 프로그램은 개별 종목 데이터만 보여줍니다. 초보자는 "이 종목이 5% 올랐는데, 이게 좋은 건지 나쁜 건지"를 판단할 기준이 없습니다.

**제안하는 개선:**
1. **시장 개요(Market Overview) 화면 신설**:
   - KOSPI / KOSDAQ 지수 현황(등락률, 거래대금)을 대시보드 상단에 항상 표시.
   - "오늘 시장: 전반적으로 하락 (-1.2%)" 처럼 한줄 시장 요약 배너 추가.
2. **벤치마크 대비 성과 비교**:
   - 모든 수익률 표시 시 KODEX 200(069500) 대비 초과 수익 여부를 함께 표시.
   - 예: "3개월 수익률 +12% (KODEX 200 대비 +5%p)"
3. **섹터 순환(Sector Rotation) 지도**:
   - 현재 어느 섹터(반도체, 바이오, 에너지 등)가 강세인지 히트맵으로 표시.
   - 주간/월간 기준으로 상위 3개 섹터와 하위 3개 섹터를 요약.

---

### 9.3 수급 분석의 깊이 부족

**현재 문제점**: 현재 수급 데이터는 **1거래일 기준** 순매수량만 표시합니다. 이것만으로는 트렌드를 파악하기 어렵습니다.

**제안하는 개선:**
1. **누적 수급 추이 차트**: 5거래일/20거래일 기간 동안 외국인·기관의 누적 순매수를 누적 막대 차트(Cumulative Bar Chart)로 시각화.
   - "최근 20일간 외국인 5,000만주 순매수" 같은 트렌드를 한눈에 파악 가능.
2. **수급 신호 표시**: 최근 5거래일 연속 외국인 순매수/순매도 발생 시 종목 카드에 아이콘 표시 (📈 외국인 연속 순매수 5일).
3. **스캐너에 수급 프리셋 추가**: "최근 5일 외국인 순매수 상위 20개" 등의 원클릭 프리셋 필터 제공.

### 9.4 뉴스 감성 분석 완성 (현재 미완성)

**현재 문제점**: `news_analyzer.py`와 `perplexity_service.py`가 존재하지만 감성 분석 기능은 실제로 구현이 미완성 상태입니다.

**제안하는 개선:**
1. **뉴스 감성 점수화**: 각 뉴스 기사에 긍정/부정/중립 감성 레이블을 부여하여 종목 상세 페이지에 "최근 뉴스 감성: 🟢 긍정 (7/10)" 식으로 표시.
2. **뉴스 이벤트 타임라인**: 가격 차트 위에 주요 뉴스 발생 시점을 마커(▲)로 표시하여 "이 날 급등의 원인이 이 뉴스였구나"를 직관적으로 파악 가능.
3. **AI 뉴스 요약 (한줄 요약)**: AI를 이용해 최근 5개 뉴스를 2~3줄로 요약한 "이번 주 핵심 뉴스" 위젯을 대시보드에 추가.

### 9.5 리스크 관리 기능 강화

**현재 문제점**: 변동성(Volatility), MDD 같은 리스크 지표가 있지만, 초보자에게 이 수치가 "나에게 어떤 의미인지" 설명이 없습니다.

**제안하는 개선:**
1. **리스크 등급 표시 (5단계)**: 변동성과 MDD를 종합하여 "안전 / 낮음 / 보통 / 높음 / 매우높음" 5단계 등급으로 표시.
   - 예: 변동성 30% 이상 → 🔴 매우 높음 (주가가 하루에 ±3% 이상 움직일 수 있음).
2. **MDD를 금액으로 표시**: MDD -30%를 "1,000만원 투자 시 최대 300만원까지 손실 가능했던 구간이 존재했습니다" 식으로 실제 금액으로 환산하여 리스크를 체감하게 함.
3. **포트폴리오 집중도 경고**: 포트폴리오 화면에서 단일 종목 비중이 40% 초과 시 "⚠️ 집중 투자 위험: {종목명}에 너무 많이 투자되어 있습니다. 분산을 권장합니다" 경고 배너 표시.
4. **상관관계 경고**: 비교 화면에서 두 종목의 상관관계가 0.9 이상이면 "이 두 종목은 매우 비슷하게 움직입니다. 두 종목을 동시에 보유해도 분산 효과가 거의 없습니다"라는 안내 표시.

### 9.6 배당 정보 부재

**현재 문제점**: ETF 투자의 핵심 장점 중 하나인 **배당(분배금)** 관련 정보가 전혀 없습니다. 특히 월배당 ETF(TIGER 미국채10년국채, KODEX 미국달러SOFR금리 등)에 대한 정보는 초보 투자자들이 가장 많이 찾는 항목입니다.

**제안하는 개선:**
1. **배당 이력 테이블**: ETF 상세 페이지에 과거 분배금 지급 이력(지급일, 지급액, 연환산 수익률)을 표시.
2. **배당 수익률(Distribution Yield) 지표 추가**: 종목 카드와 스캐너에 연 배당수익률을 표시.
   - 예: "연 배당수익률: 4.2%"
3. **배당 캘린더**: 월별 분배금 지급 예정일을 달력 형태로 표시하는 화면 추가.
   - 데이터 소스: 네이버 금융의 ETF 분배금 이력 페이지 스크래핑.

### 9.7 스캐너(종목 발굴) UI/UX 개선

**현재 문제점**: 스캐너는 강력한 기능을 가지고 있지만, 초보자가 "어떤 조건으로 검색해야 할지" 알기 어렵습니다.

**제안하는 개선:**
1. **초보자용 프리셋 필터 추가**:
   - "📗 입문자 추천": 거래량 상위 + 낮은 변동성 + 양의 YTD 수익률
   - "🛡️ 안정형": MDD -15% 미만 + 변동성 하위 30%
   - "🚀 성장형": YTD 수익률 상위 + 외국인 순매수
   - "💰 배당형": 배당수익률 3% 이상 (배당 데이터 연동 후)
2. **결과 화면에 "이 ETF에 투자해야 하는 이유" 한줄 설명 추가**: AI 인사이트를 활용하여 스캐너 결과 각 종목 옆에 짧은 코멘트 자동 생성.
   - 예: "반도체 업종 강세 수혜, 외국인 5일 연속 순매수"

### 9.8 투자 일지(Trading Journal) 기능

**현재 문제점**: 프로그램이 가상의 포트폴리오 분석은 지원하지만, **실제 매수/매도 거래를 기록하고 추적**하는 기능이 없습니다. 초보자는 자신의 매매 패턴을 돌아보는 경험이 성장에 매우 중요합니다.

**제안하는 개선:**
1. **매매 기록 입력 폼**: 날짜, 종목, 매수/매도, 수량, 단가를 입력.
2. **매매 이력 캘린더**: 월별 달력에 매수(파랑)·매도(빨강) 거래일을 마킹하여 시각적으로 투자 패턴 파악.
3. **실현 손익 계산**: 매도 시 실현 수익률과 손익 금액을 자동 계산.
4. **투자 메모**: 각 거래마다 매매 이유를 메모로 남겨 나중에 "왜 그때 샀지?"를 돌아볼 수 있는 회고 기능.

### 10.1 보안 취약점 (Security Vulnerabilities)

#### [CRITICAL] ⚠️ 중복 발견 — .env 파일에 실제 API 키 하드코딩 + 프론트엔드 빌드 결과물에 API 키 노출
> **두 분석에서 동일하게 발견됨** (research.md 분석 + research2.md §1.1)

- **파일**: `.env`, `frontend/src/services/api.js`, `.env.example`, `render.yaml`
- **위험 항목**:
  - `API_KEY` (백엔드 인증 키) — `.env`에 평문 기록, Git 히스토리에 영구 노출
  - `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET` — 동일
  - `VITE_API_KEY` — Vite 빌드 시 번들에 포함되어 브라우저 개발자 도구에서 누구나 확인 가능
- **심각도**: CRITICAL
- **위험**:
  - Git 히스토리에 영구 기록되어 저장소 접근 권한이 있는 누구나 키를 확인할 수 있음
  - `VITE_API_KEY`는 데이터 수집 트리거·설정 변경 등 민감한 백엔드 권한을 가지고 있어, 브라우저 노출 시 누구나 관리 API를 호출할 수 있음
- **수정 방안**:
  1. `.env`를 즉시 `.gitignore`에 추가하고 노출된 모든 키를 재발급
  2. 커밋에는 `.env.example`(실제 값 없음)만 포함
  3. 민감 작업(수집 트리거, 설정 변경 등)은 프론트엔드에서 API 키를 직접 전송하지 않고, 별도 관리자 인증(OAuth2, Session/Cookie)으로 분리
  4. 현재 구조를 유지해야 한다면, `VITE_API_KEY` 권한을 읽기 전용으로 분리하거나 IP 화이트리스팅을 추가 적용

#### [HIGH] 스캐너 엔드포인트의 동적 컬럼명 SQL 삽입 위험
- **파일**: `backend/app/routers/scanner.py`, 약 163·172–175번째 줄
- **문제**: `sort_column`, `sort_dir_sql` 값을 f-string으로 직접 SQL에 삽입
  ```python
  f"ORDER BY {sort_column} {sort_dir_sql}"
  ```
- **심각도**: HIGH
- **위험**: 컬럼명은 파라미터 바인딩 대상이 아니라 화이트리스트 검증이 필요함. 허용된 컬럼명 목록과 대조하지 않으면 SQL Injection 가능
- **조치**: 허용 컬럼명 목록(`ALLOWED_SORT_COLUMNS = [...]`)을 정의하고, 입력값이 목록에 없으면 기본값으로 치환

#### [MEDIUM] ⚠️ 중복 발견 — 개발 모드에서 API 키 인증 우회 (Fail-open)
> **두 분석에서 동일하게 발견됨** (research.md 분석 + research2.md §1.2)

- **파일**: `backend/app/middleware/auth.py`, 117–128번째 줄
- **문제**: `API_KEY` 환경 변수 미설정 시 모든 요청을 인증 없이 허용하고 `"dev-mode"` 문자열 반환. 특정 환경 변수(`RENDER` 등)로 프로덕션을 감지하지만, 감지에 실패하면 프로덕션에서도 인증이 무력화됨
- **심각도**: MEDIUM
- **위험**: `API_KEY` 미설정 상태로 배포 시 인증이 완전히 무력화 (Fail-open 구조)
- **수정 방안**:
  1. 기본 동작을 Fail-safe(차단)로 전환: `API_KEY` 미설정이면 요청을 거부
  2. 개발 환경임을 명시적으로 선언하는 변수(`APP_ENV=development`)가 있을 때만 인증 우회 허용
  3. FastAPI `startup` 이벤트에서 `APP_ENV=production`이고 `API_KEY`가 없으면 서버 시작 자체를 거부하도록 체크 추가:
  ```python
  # main.py startup event
  if os.getenv("APP_ENV") == "production" and not os.getenv("API_KEY"):
      raise RuntimeError("API_KEY must be set in production environment")
  ```

#### [MEDIUM] 내부 오류 정보 외부 노출 (Information Leakage)
- **파일**: `backend/app/routers/data.py` 등 여러 파일
- **문제**: 예외 메시지를 그대로 HTTP 응답에 포함
  ```python
  raise HTTPException(status_code=500, detail=f"DB error: {e}")
  ```
- **심각도**: MEDIUM
- **위험**: DB 스키마, 파일 경로, 쿼리 구조 등 내부 구현 정보가 외부에 노출될 수 있음
- **조치**: 사용자에게는 일반적인 오류 메시지만 반환하고, 상세 오류는 서버 로그에만 기록

#### [MEDIUM] 민감 작업에 대한 감사 로그(Audit Log) 부재
- **파일**: `backend/app/routers/settings.py`, `backend/app/routers/data.py`
- **문제**: 종목 삭제(`DELETE /api/settings/stocks/{ticker}`), DB 초기화(`DELETE /api/data/reset`) 등 돌이킬 수 없는 작업에 대해 누가 언제 실행했는지 기록이 없음
- **심각도**: MEDIUM
- **조치**: 이러한 작업 수행 시 `audit_log` 테이블에 타임스탬프, IP, 작업 내용을 기록

### 10.3 코드 품질 개선 사항 (Code Quality)

#### [MEDIUM] N+1 쿼리 패턴 — 수집 상태 엔드포인트
- **파일**: `backend/app/routers/data.py`, `get_collection_status()` 함수
- **문제**: 등록된 모든 종목을 순회하며 종목별로 개별 DB 쿼리 실행
  ```python
  for ticker in etfs:
      data = collector.get_price_data(ticker, ...)  # DB hit per ticker
  ```
- **심각도**: MEDIUM (종목 수 증가 시 응답 지연 심화)
- **조치**: 단일 `IN (...)` 쿼리 또는 배치 조회로 리팩터링

#### [MEDIUM] 과도하게 넓은 예외 처리 (`except Exception`)
- **파일**: `backend/app/routers/data.py` 173번째 줄, `backend/app/database.py` 463번째 줄, `backend/app/services/ticker_catalog_collector.py` 외 다수
- **문제**: `except Exception:` 으로 모든 예외를 묵살하여 운영 중 오류 원인 파악 불가
  ```python
  except Exception:
      pass  # 또는 무의미한 fallback
  ```
- **조치**: 예상 가능한 예외 타입(`sqlite3.OperationalError`, `httpx.TimeoutException` 등)을 명시적으로 처리하고, 예상치 못한 예외는 반드시 로깅

#### [MEDIUM] 티커 파라미터 형식 검증 부재
- **파일**: `backend/app/routers/etfs.py` 외 다수 라우터
- **문제**: URL 경로에서 받는 `{ticker}` 파라미터에 대해 형식 검증(영문/숫자 6자리 이내 등) 없이 DB 쿼리에 사용
- **조치**: 정규식 패턴 검증 추가 (`^[A-Z0-9]{3,6}$` 또는 한국 종목코드 형식)

#### [MEDIUM] API 키 저장 위치 이중화로 인한 상태 불일치
- **파일**: `.env` (환경 변수), `backend/config/api-keys.json` (파일 저장)
- **문제**: `PUT /api/settings/api-keys`로 키를 변경하면 `api-keys.json`과 `os.environ`만 갱신되고 `.env`는 변경되지 않아 서버 재시작 시 `.env` 값으로 덮어씌워짐
- **조치**: 단일 저장 위치(환경 변수 또는 DB/파일 중 하나)로 통일

#### [LOW] 타입 힌트 누락
- **파일**: `backend/app/routers/settings.py` 202번째 줄 외 다수
- **문제**: `stock_data: StockUpdate = None` — `Optional[StockUpdate]` 또는 `StockUpdate | None`으로 명시해야 함
- **조치**: `from typing import Optional` 사용 또는 Python 3.10+ 유니온 문법 적용

#### [LOW] 프론트엔드 — 환경 변수 API 키 브라우저 노출
- **파일**: `frontend/src/services/api.js`
- **문제**: `import.meta.env.VITE_API_KEY`는 Vite 빌드 결과물에 평문으로 번들링되어 브라우저 개발자 도구에서 확인 가능
- **조치**: 민감 작업은 백엔드 프록시를 통해 처리하거나, 프론트엔드용 키와 백엔드용 키를 분리하여 프론트 키의 권한 범위를 최소화
